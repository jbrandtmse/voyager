# ADR 0020 — MADR Format for ADRs; `docs/adr/` Location

Status: Accepted
Date: 2026-05-18
Deciders: Voyager project maintainer

## Status

Accepted. All ADRs follow the MADR (Markdown Architectural Decision Records) minimal 5-section format. ADRs live under `docs/adr/` with filenames `NNNN-kebab-case-title.md`.

## Context

NFR-M1 mandates "an ADR per rejected technical idea" and NFR-M6 commits to hands-off-maintainability. Both depend on ADRs being a stable, predictable, AI-agent-readable archive that future-self and downstream agents can navigate without re-deriving conventions.

The ADR-tooling ecosystem in 2026 standardizes on MADR (https://adr.github.io/madr/), now at version 4.x. Two MADR variants exist: the minimal 5-section (status / context / decision / consequences / alternatives) and the extended variant with additional sections (decision drivers, considered options, pros-and-cons of the options, links). The minimal variant matches the project's terse-ADR preference and matches what AI agents have the easiest time consuming.

## Decision

**Format: MADR minimal 5-section variant.** Every ADR has, in order:

1. `## Status` — one of: `Proposed`, `Accepted`, `Deprecated`, `Superseded-by-NNNN`.
2. `## Context` — the problem, constraints, forces, cited sources.
3. `## Decision` — what we chose.
4. `## Consequences` — positive, negative, obligations.
5. `## Alternatives Considered` — rejected options with brief evaluation.

**Storage:** `docs/adr/NNNN-kebab-case-title.md` where `NNNN` is a 4-digit zero-padded sequence number. No subdirectories.

**Template:** `docs/adr/0000-template.md` is the canonical template, numbered `0000` so the index regenerator picks it up uniformly.

**Index:** `docs/adr/README.md` is auto-generated by `scripts/adr-index.py` (eventually wrapped by a `just adr-index` recipe per Story 1.4). The script scans the directory, parses each file's H1 title and `Status:` line, and writes a table of contents sorted by ADR number. README.md is not hand-edited.

**Immutability:** ADRs are immutable once accepted. Updates produce a *new* ADR with `Status: Superseded-by-NNNN` linking to the successor. Errata to an accepted ADR (typo fixes, broken-link repairs) are permitted; substantive content changes require a successor.

**When an ADR is required:**

- Every entry in the technical research's "Rejected Technical Ideas" inventory (NFR-M1).
- Every open-question decision made in the architecture workflow with real alternatives.
- Every infrastructure or CDN-level decision.
- Every change to the URL contract or other versioned public-API surface (ADR 0001).
- New ADRs added during implementation when a substantive trade-off is made.

## Consequences

**Positive:**
- AI agents (Claude Code, future-self) navigate the archive predictably.
- Industry-standard format means contributors don't have to learn project-specific conventions.
- The index is greppable, sortable, machine-checkable (and tested — see `bake/tests/test_adr_catalogue.py`).
- Filename numbering forces a global sequence; collisions are impossible by construction.

**Negative:**
- The minimal MADR variant lacks explicit "Decision Drivers" and "Pros/Cons of the Options" sections; these are folded into Context and Alternatives respectively. Mitigated: the project's ADRs cite source documents heavily, so the drivers live in the cited material.
- ADR immutability means errors-in-spirit (not just typos) require a fresh successor ADR with the supersession header. Cost-of-ceremony is real but small.

**Obligations on downstream stories:**
- Every story that introduces a substantive trade-off authors an ADR (or amends a deferred Proposed ADR like 0016).
- Story 1.4 will add `just adr-index` recipe wrapping `python scripts/adr-index.py`.
- The `bake/tests/test_adr_catalogue.py` test enforces section-header completeness and status-value validity.

## Alternatives Considered

- **MADR extended (full template).** Rejected: the additional sections (decision drivers, options matrix, pros/cons) over-formalize for a solo-developer portfolio project; the minimal variant matches the project's terse-ADR preference.
- **Nygard's original ADR format.** Rejected: pre-dates MADR; less section-disciplined; less tooling support in 2026.
- **OADR / Y-statement / "consequences-first" formats.** Rejected: each has merit but none has the ecosystem tooling MADR has.
- **ADRs as PRs with `## Architectural Decision` headings inside.** Rejected: PRs scroll out of history; ADRs need a stable in-repo location for grep and link.
- **Subdirectories per category** (e.g., `docs/adr/rendering/`, `docs/adr/build/`). Rejected: the global-sequence numbering plus the index table-of-contents gives all the discoverability a category folder would; ADRs are cross-cutting enough that any taxonomy would be contested.
