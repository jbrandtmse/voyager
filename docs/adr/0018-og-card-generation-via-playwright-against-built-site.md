# ADR 0018 — OG Card Generation via Playwright Against Built Site

Status: Accepted
Date: 2026-05-18
Deciders: Voyager project maintainer

## Status

Accepted. Pre-rendered OG (Open Graph) card images are generated by a dedicated Playwright spec against the built site, reusing the L4 visual-regression harness.

## Context

FR39 requires pre-rendered OG cards for each chapter so that sharing a deep-link surfaces a meaningful preview image on Twitter, Discord, Slack, iMessage, etc. (these clients fetch `og:image` via HTTP and never execute JavaScript). The card must show the canonical frame of the chapter — e.g., Voyager 1 approaching Jupiter — at 1200×630 px.

Two architectural shapes are possible:

1. **Author each card by hand** — designer crops a frame in a paint program, commits PNGs. High labor, drifts from the live render, doesn't update when scenes evolve.
2. **Generate from the live render** — automation visits each chapter at its anchor ET and screenshots the canvas. Single source of truth between the OG card and what users actually see.

Within option 2, the tooling choice matters: a headless browser is required (Three.js needs WebGL), and the L4 visual-regression layer in the test pyramid already uses Playwright. Reusing that harness for OG cards eliminates duplicate infrastructure.

## Decision

**Generate OG cards via Playwright against the built site.**

- A dedicated spec file `e2e/og-cards.spec.ts` navigates to each chapter URL with a deterministic `?ogCard=true` flag.
- The `?ogCard=true` flag fixes the simulation at the chapter anchor ET, hides chrome, and applies any canonical-frame staging.
- Playwright screenshots at 1200×630 → `web/public/og/<chapter-id>.png` (or `web/dist/og/<chapter-id>.png` post-build).
- CI runs this step after `vite build` and *before* `playwright test` (the L4 visual-regression run). L4 then validates the live site against the same canonical frames used in OG cards — single source of truth.
- The `og/` directory is included in the deployed bundle; static `<meta property="og:image">` tags in each chapter's pre-rendered HTML shell reference it.

## Consequences

**Positive:**
- OG cards and L4 visual regression share the same canonical frame definitions — they cannot drift apart.
- Scene evolution automatically refreshes OG cards on the next build; no manual export step.
- Playwright is already required for L4 + L5; no new tool to learn or maintain.
- Pre-rendered images (not runtime canvas) means social media crawlers see them immediately — no waiting on JS execution.

**Negative:**
- The `?ogCard=true` flag is a small but real coupling between rendering code and the OG card pipeline. Mitigation: it's a strict-boolean URL param like `?embed=true` (NFR-S7), with the same parsing discipline.
- Build time grows by the screenshot pass (~30 s for 11 chapters). Within the NFR-M4 ≤15 min slow-tier budget.

**Obligations on downstream stories:**
- Epic 7 / Story 7.x implements `e2e/og-cards.spec.ts` and the `?ogCard=true` rendering hook.
- The L4 visual regression spec consumes the same chapter list and reuses the canonical-frame staging.
- Each chapter's pre-rendered HTML shell (Vite multi-page input) emits `<meta property="og:image" content="/og/<chapter-id>.png">`.
- Chrome-DevTools MCP (per ADR 0010) is *not* used for OG cards — Playwright is the CI-time tool.

## Alternatives Considered

- **Hand-authored PNGs in a `public/og/` directory.** Rejected: labor cost compounds with each scene tweak; OG cards silently drift from the live render; future-self will not maintain manual exports.
- **Puppeteer (instead of Playwright).** Rejected: Playwright is already required for L4 + L5; using two headless-browser drivers doubles maintenance.
- **Server-side rendering via Satori (Vercel's React-to-SVG renderer).** Rejected: cannot render Three.js scenes; couples to Vercel ecosystem; would force the OG card visual style to diverge from the canvas (the whole point is *don't* let them diverge).
- **Skip OG cards entirely; rely on the chapter HTML's `<title>` and meta description.** Rejected: FR39 explicitly requires preview images; social-media share previews are a major discoverability surface.
- **Generate cards at runtime via a serverless function.** Rejected: violates the no-server commitment (NFR-Sc1, NFR-S9); introduces a runtime dependency on a cold-start function for every social share preview.
