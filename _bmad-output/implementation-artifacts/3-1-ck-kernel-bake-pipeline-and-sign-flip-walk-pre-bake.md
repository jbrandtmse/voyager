# Story 3.1: CK Kernel Bake Pipeline and Sign-Flip Walk Pre-Bake

**Epic:** 3 — Attitude Reconstruction (the Differentiator)
**Status:** review
**Date created:** 2026-05-20
**Source:** epics.md § Epic 3 Story 3.1 (lines 1270–1304); ADR-0024 (pre-bake quaternion sign-flip walk); ADR-0004 (VTRJ binary format); ADR-0011 (Git LFS kernel storage)

## User Story

As the project maintainer,
I want the Phase 0 CK kernels (vgr1_super_v2.bc, vgr2_super_v2.bc, plus PDS Rings Node ISS SEDR supplements) extracted into deterministic VTRJ attitude binaries with quaternion sign-flip discontinuities pre-walked into a continuous representation,
So that the web-side `AttitudeService` (Story 3.2) can SLERP-interpolate without sign-flip artifacts at sample knots, Story 3.4 can apply the attitude per-frame to both spacecraft, Story 3.6 can drive the `<v-attitude-indicator>` HUD provenance element from a single trusted substrate, and Story 3.7's L2 JS-vs-SPICE attitude consistency gate has bake outputs to validate against — fulfilling FR15 / FR16 / FR17 / FR20 baseline.

### AC1-T1 — VTRJ schema extension for quaternions (sub-AC, schema decision required)

The existing `vtrj_writer.write_vtrj(body_id, et_start, et_end, cadence_seconds, samples)` enforces `samples.shape == (N, 6)` (state vector) and `body_id ∈ ALLOWED_BODY_IDS = {-31, -32, 10, 1..8, 301}`. Story 3.1 needs `(N, 4)` quaternion samples and `body_id ∈ {-31000, -31100, -32000, -32100}` (CK structure IDs per `bake/src/ck_inventory.py:36-43`). The dev MUST choose one of:

- **Path (a) — Extend `vtrj_writer.py` in place.** Make `write_vtrj` accept `(N, 6)` OR `(N, 4)` samples; widen `ALLOWED_BODY_IDS` to include the CK structure IDs; add a `kind: Literal["trajectory", "attitude"]` parameter (or infer from shape — explicit parameter preferred for ADR-0004 clarity). Update `read_vtrj` symmetrically. **ADR-0004 is already structured to allow `componentsPerSample` variation per ADR-0024 § Decision** ("Then serialize to VTRJ (ADR 0004 with `componentsPerSample=4`)"), so this is the ADR-blessed path.
- **Path (b) — New sibling `vtrj_attitude_writer.py`.** Keeps `vtrj_writer.py` shape-locked at `(N, 6)`. New writer shares the 40-byte header struct (or extends it) but enforces `(N, 4)` and the CK structure IDs. Pro: tidy separation. Con: read_vtrj also needs splitting (or a discriminated-union read), which is more code surface for marginal gain.

**Strong recommendation: path (a).** ADR-0024 explicitly contemplates VTRJ-with-componentsPerSample=4. Document the schema extension in `vtrj_writer.py`'s module docstring (preserve the existing locked-by-Story-1.4 AC2 doc block, add a "Story 3.1 schema extension" section). The dev MUST update `ALLOWED_BODY_IDS` to add `{-31000, -31100, -32000, -32100}` AND must add a new defense test in `bake/tests/test_vtrj_writer.py` covering the attitude path (write a `(N, 4)` array with `body_id=-31000`, round-trip read it, assert quaternion values match).

The chosen path is documented in `Completion Notes List` (and the alternative explicitly noted as rejected with rationale).

### AC2 — Quaternion sign-flip walk (epics.md AC2, ADR-0024 obligation)

- **GIVEN** raw CK output may contain quaternion sign flips (consecutive samples have opposite signs representing the same rotation — happens when SPICE's internal evaluation crosses a sign boundary; ADR-0024 § Context)
- **WHEN** the sign-flip walk pre-bake runs
- **THEN** the bake module `bake/src/quat_continuity.py` exports a function (e.g., `walk_signs(quats: np.ndarray) -> np.ndarray`) that traverses each quaternion stream in time order and negates samples whose dot product with the prior sample is negative (per ADR-0024 § Decision pseudocode)
- **AND** the walked output guarantees `np.dot(q[i], q[i+1]) >= 0` for every consecutive pair (continuous representation suitable for SLERP, no shortest-path branch needed at runtime)
- **AND** the walk is byte-deterministic on identical input (no randomness, no host-clock dependency)
- **AND** `quat_continuity.py` is callable from both `ck_sample.py` (as the pre-VTRJ-write transform) and from the L1 validator (for the synthetic-adversarial test below)
- **AND** a unit test in `bake/tests/test_quat_continuity.py` verifies:
  - A synthetic adversarial input where every other sample is sign-flipped (`q[i] = -q[i]` for odd `i`) is walked back to a continuous stream (`dot(q[i], q[i+1]) >= 0` for all consecutive pairs)
  - An already-continuous input passes through unchanged (`np.array_equal(input, walked)`)
  - Edge case: empty array, single-element array — return the input unmodified (no out-of-bounds index)
  - Edge case: an exactly-zero dot product (orthogonal quaternions, which happens for 180° rotations) — choose deterministic tie-break: `dot < 0` flips, `dot >= 0` does not — document the choice in the module docstring

### Closest-approach precedence in the ET grid

For each encounter window, the ET grid is built by concatenating three bands with explicit precedence:

```
1. 10-second band:  [closest_approach_et - 3600, closest_approach_et + 3600]
2. 1-minute band:   [encounter_start_et - 172800, encounter_end_et + 172800]
3. daily band:      [the CK coverage interval, masked to remove (1) and (2)]
```

Generate (3) first, mask out (2)'s range, mask out (1)'s range from the masked (2). Then concatenate in time order. Document the precedence + the masking algorithm in `ck_sample.py`'s module docstring.

### Quaternion convention disambiguation

SpiceyPy `ckgp` returns a quaternion in SPICE convention. The SPICE convention is `[cos(θ/2), sin(θ/2) * axis_x, sin(θ/2) * axis_y, sin(θ/2) * axis_z]` — i.e., scalar-first `[w, x, y, z]` storage. The Three.js / WebGL runtime convention is scalar-LAST `[x, y, z, w]`. Story 3.2 (AttitudeService) does the conversion at decode time. Story 3.1's VTRJ binary stores in **SPICE convention** (`[w, x, y, z]`) — document this in the schema-extension docstring AND in the manifest entry's metadata so the runtime decoder knows.

**Quaternion convention disambiguation (documented in two places).**
SpiceyPy `m2q` (and the SCM-to-quaternion contract) returns SPICE scalar-first convention `[w, x, y, z]`. Bake-side VTRJ binaries store in this exact ordering. SciPy `Rotation.from_quat` / `as_quat` uses scalar-LAST `[x, y, z, w]`, so the L1 validator at `_slerp_at_ets` does a column-swap on input and reverses it on output. The runtime decoder (Story 3.2 `AttitudeService`) will do the same scalar-last permute at decode time when handing off to `THREE.Quaternion`. This is documented in `vtrj_writer.py`'s "Story 3.1 schema extension" docstring section AND in `ck_sample.py`'s module docstring.

**ck_sample sampling primitive: `ckgp` (pointing-only) not `ckgpav` (pointing+av).**
The bake doesn't need angular velocity — only orientation — so `ckgp` is the minimal primitive. `ckgp` + `m2q` (matrix-to-quaternion) is the direct path. Documented inline in `sample_window_pointing_only`. Both helpers (`sample_window_with_mask` using `ckgpav` and `sample_window_pointing_only` using `ckgp`) are present in the module; the orchestrator uses the pointing-only path.

**SCLK encoding: SC ID derived via truncate-toward-zero (`-(abs(struct_id) // 1000)`).**
SpiceyPy's `sce2c` encodes ET to SCLK ticks per spacecraft NAIF SPK ID (-31 / -32), not CK structure ID. During dev I initially used Python floor-division `struct_id // 1000` — but Python's `//` on negatives floors toward negative infinity, so `-31100 // 1000 = -32` (WRONG — V1 scan-platform should route to V1 SCLK, not V2). Caught this in the Completion Notes review pass and replaced with `-(abs(struct_id) // 1000)` (truncate-toward-zero) in three call sites: `ck_sample.sample_window_with_mask`, `ck_sample.sample_window_pointing_only`, and `validate_l1._ckgp_quaternions_at_ets`. `-(abs(-31100) // 1000) = -(31100 // 1000) = -(31) = -31` ✓ for V1 scan platform; same form gives `-31` for V1 bus (`-31000`), `-32` for V2 bus (`-32000`), and `-32` for V2 scan platform (`-32100`). Locked by two new regression tests in `test_ck_sample.py`: `test_sc_id_derivation_truncates_toward_zero` (parametric check across all 4 CK structures + asserts floor-div gives the wrong answer for the scan-platform IDs to prove the test would catch a regression) and `test_ck_sample_source_uses_truncate_toward_zero_for_sc_id` (source-level grep tripwire that strips backticked comment spans before checking for bare `struct_id // 1000` in code). Fast-tier tests caught this bug at code-review time (before slow-tier kernel-dependent tests would have surfaced it at LFS-CI time).

### Lead-driven smoke evidence + design pivot (2026-05-21)

The lead-driven slow-tier smoke (the AC8 Integration AC against real LFS-pulled CK kernels) surfaced TWO HIGH-severity defects that the fast tier had masked:

1. **`spice.ckgp` / `spice.ckgpav` 3-tuple unpack vs SpiceyPy 8.1.0's 2-tuple/3-tuple return.** Both `sample_window_with_mask` and `sample_window_pointing_only` in `ck_sample.py` and the validator's `_ckgp_quaternions_at_ets` in `validate_l1.py` unpacked `(cmat, _clkout, found)` from `ckgp`. SpiceyPy 8.1.0 with `found_check=True` (default) returns just `(cmat, clkout)` and RAISES `SpiceyError` on not-found. The dev's unit tests mocked the SpiceyPy calls with 3-tuple returns, so all 22 + 13 + 13 + 9 fast-tier tests passed against a fictitious API. Real SpiceyPy raised `ValueError: not enough values to unpack` on every attitude sample.

   **Fix (lead-side, 2026-05-21):** all three callsites unpack 2-tuple for `ckgp` / 3-tuple `(cmat, av, clkout)` for `ckgpav`; the `if not found: continue` lines (which would have errored on the missing `found` variable) are dropped — the `try/except SpiceyError` ALREADY handles the not-found case correctly.

2. **Mixed-cadence VTRJ serialization with `linspace` knot reconstruction was mathematically incoherent.** The dev shipped a bake that grabbed CK samples at 10-sec / 1-min / daily mixed cadences across the FULL CK coverage (1977-2030) per encounter file, then stored only `et_start` + `et_end` + `sample_count` in the VTRJ header. The runtime decoder (and the L1 validator) reconstructed sample positions via `np.linspace(et_start, et_end, N)` — UNIFORM spacing. With dense samples near closest approach + sparse samples in cruise, the linspace reconstruction puts knots at the wrong ETs by hundreds of millions of seconds, producing SLERP errors of up to π radians (~180°) against the canonical ckgp output. The fast-tier mocked ckgp returns made this invisible; real SpiceyPy proved errors of 2800-3140 mrad.

   **Fix (lead-side, 2026-05-21, user-confirmed Option 3 design pivot):**
   - **ADR-0004 amended** with a "Body Layout per Kind" section (Story 3.1 amendment). The VTRJ header struct (`<4sHiddId2s`) is unchanged; the BODY shape is now discriminated by `body_id` namespace:
     - Trajectory body IDs (`{-31, -32, 10, 1..8, 301}`): body = `(N, 6)` `[x,y,z,vx,vy,vz]`, uniform cadence, time implicit.
     - Attitude body IDs (`{-31000, -31100, -32000, -32100}`): body = `(N, 5)` `[et, qw, qx, qy, qz]`, explicit per-sample ET (column 0), `cadence_seconds` header field informational only.
   - **`vtrj_writer.py`** extended: `write_vtrj(kind="attitude", ets=...)` interleaves explicit ETs into the (N,5) body. `read_vtrj` returns (N,5) for attitude kind. BYTES_PER_SAMPLE_BY_KIND updated (32 → 40 for attitude).
   - **`ck_sample.bake_attitude`** passes the explicit kept-ET array to `write_vtrj`.
   - **`validate_l1.py`** reads explicit ETs from column 0 of the (N,5) body for SciPy Slerp knots.
   - **Architecture.md** and **PRD** updated to reflect the explicit-ET attitude path.
