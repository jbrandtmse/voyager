## What went well (user-prioritised)

**🏆 Top win: Defense-test discipline at each smoke defect.**

Across Stories 4.3 + 4.5, the per-story smoke surfaced 4 test-pyramid-invisible defects. Each defect prompted a focused dev re-spawn that:

1. Fixed the immediate bug with minimal-change discipline.
2. Added defense tests at the unit-test tier specific to the failure mode (not generic "feature works" tests).
3. Documented the trigger pattern so similar future bugs become recognisable.

The cumulative ~130 defense tests across 4.3 / 4.5 / 4.6 / 4.7 mean a future regression of any of these defects now fails at the unit-test tier instead of needing another smoke iteration. This is exactly the closure the /epic-cycle workflow's Lesson 8 names: "the test pyramid is necessary but not sufficient — the per-story smoke is the bridge from 'tests pass' to 'the product works,' and the defense tests added at each smoke defect are the load-bearing closure."

**Other wins worth recording:**

- **Pattern reuse 4.5 → 4.6 → 4.7.** Story 4.5 established the `applyDefaultFraming` subscriber + cold-load replay pattern. Stories 4.6 (3 chapters) + 4.7 (2 chapters) shipped clean on iter-1 with zero new wire-up code. The pattern is now load-bearing for future encounter chapters.
- **Rule 5 amendment discipline (6 in-place amendments).** Each amendment cited rationale in spec docstrings + Dev Agent Records, not buried in comments or `deferred-work.md`. Planning artifacts stayed honest.
- **Manual-review bug-fix sweep closure.** 8 bugs triaged + closed with self-contained audit trail (bug-report folder now has `## Closure` sections on every file). 2 misfilings caught with primary-source evidence.

## What didn't go well (user-prioritised)

**🚨 Top concern: Satellite SPK kernel procurement deferred.**

Story 4.3 built moon meshes for all 12 Voyager-encounter satellites (Io / Europa / Ganymede / Callisto for V1J+V2J; Titan / Iapetus / Hyperion for V1S+V2S; Miranda / Ariel / Umbriel / Titania / Oberon for V2U; Triton for V2N). Each mesh has a 2K KTX2 texture and renders correctly. But moon meshes are invisible at chapter cold-load because their position chunks haven't been baked — the satellite SPK kernels (`jup365.bsp`, `sat427.bsp`, `ura111.bsp`, `nep097.bsp`) aren't in `kernels/kernels-manifest.json`. Smoke evidence from Stories 4.5 + 4.6 + 4.7 shows all moon meshes stacked at the gas-giant's heliocentric coordinates (their fallback position when no ephemeris is available).

This means a sighted user navigating to `/c/v1-jupiter` sees Jupiter at the centered origin + V1 spacecraft nearby, but the Galilean moons that V1 actually flew past are not visually present. The encounter feels less complete than it should.

The decision to defer to a follow-up (per the bake-side `MOON_BODIES` table with graceful `SpiceyError` skip) was the right call at story time — procurement scope would have ballooned 4.3 further. But it's a real gap.

**Other concerns worth recording:**

- **Story 4.3 procurement detour cost.** 4.3 ran 7 dev cycles. The texture procurement (8K-down-to-4K Rule 5 amendment + 12 moon textures via Steve Albers / NASA SVS / Wikimedia) was a half-session detour. Could have been scoped tighter at sprint-planning if the procurement risk was forecast.
- **Test-pyramid-invisible defects (4 caught by smoke).** Despite the test pyramid being healthy (3075 tests by end of epic), four wire-up defects shipped green through unit + integration tests because each layer mocked the offending code path. The pattern is now known and the defense tests close the loop, but the test-design gap exists — integration tests should exercise more of the production wire-up chain by default.
- **Heliocentric system-view camera mode missing.** Story 4.8 AC1 scope-reduced. V1S Titan-slingshot bend + V2N Triton-bend canonical FR12 screenshots deferred.

## Lessons applied + new lessons surfaced

**Epic 3 lessons that paid off in Epic 4:**

- Rule 5 in-place amendments (Story 4-3 amended AC4 + AC5 + ADR-0006; Story 4-6 amended PRD V2J sequence; Story 4-8 amended AC1).
- Rule 10 Lit declare+ctor pattern (Story 4-4 reactive props; bug-fix sweep HUD components).
- Rule 11 build-pipeline E2E tests (Story 4-3 toktx build pipeline E2E).
- Lesson 8 per-story smoke (4 defects caught + closed).

**New lessons from Epic 4:**

1. **Wire-up trigger patterns are a distinct trap class.** Stories 4.3 (cold-load replay) and 4.5 (applyDefaultFraming subscriber) both shipped with the resolver REGISTERED but the AUTO-TRIGGER missing. Unit tests proved the resolver was correct; integration tests mocked the trigger; smoke caught the gap. Defense pattern: any new service that registers a resolver/handler MUST have a test that confirms the trigger fires under realistic conditions (not just that the resolver returns the right value when called directly).
2. **Pattern generalisation is the load-bearing closure.** Story 4.5 established the chapter-activation framing pattern; Stories 4.6 + 4.7 reused it cleanly with ZERO new wire-up. This says the pattern is good. Encode such patterns explicitly in the next epic's planning artifact so they're discoverable without spelunking through prior stories.
3. **Procurement detours are a planning-time smell.** Story 4.3's 7-cycle journey was dominated by procurement (8K→4K Rule 5 amend; 12 moon textures via various public-domain sources; satellite SPK kernels deferred). When a story's scope includes "procure external assets" it warrants more careful scoping at sprint-planning, including an explicit "procurement gate at task X" callout.

### Top win (user-prioritised)

**🏆 Story 4-11 procurement smoother than Story 4-3.**

Satellite SPK procurement (canonical NAIF kernels with well-known URLs + SHA verification flow) was vastly smoother than Story 4-3's moon-texture procurement (which required cross-source curation across Steve Albers / NASA SVS / Wikimedia / Johnston's Archive). Story 4-3 ran 7 dev cycles, dominated by procurement; Story 4-11 ran 1 dev cycle including procurement + bake + tests + smoke.

**The signal:** procurement stories with well-defined canonical sources are 1-cycle work; cross-source procurement is the multi-cycle expensive case. A future story scoping should treat "we know exactly which file to download from where" as a fundamentally different kind of procurement than "we need to curate from multiple sources with different licensing."

### Top concern (user-prioritised)

**🚨 Story 4-11 satellite SPK is 2.3 GB LFS (5× the story's 100-500 MB estimate).**

The `ura184_part-3.bsp` file alone is 387 MB; the four-kernel total reached ~2.3 GB. Clone-time impact uses `git lfs skip-smudge` but that's not the default for first-time contributors. A documented LFS-size policy + opt-in fetch pattern is needed.

### Cross-cutting observations from the 3 stories

1. **Pattern reuse continues to work.** Story 4.12 extracted `_applyFraming(target, animated)` as a shared private helper on `VoyagerCameraController` — `applyDefaultFraming` (Story 4.5) + `applyHeliocentricFraming` (Story 4.12) both delegate. Epic 5 PBD will inherit the same helper. The pattern is now generalised across 3 use cases (R-key restore, chapter activation, heliocentric URL).

The Playwright suite caught it because it runs against built artifacts. This is a gap NOT predicted by the main retro: tests at the dev-server level can pass while the production build is broken.
