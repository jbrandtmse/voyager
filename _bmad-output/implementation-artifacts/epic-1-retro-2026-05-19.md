## 1. Epic Outcome — Honest Summary

CI is green. All 14 stories passed code review. The product ships to Cloudflare Pages on every push to main. **And the core feature — "watch Voyager fly" — does not work in a real browser.**

Manual browser verification, performed immediately before this retrospective, surfaced four HIGH defects and one MED defect that every automated tier (Vitest unit, Vitest integration, Playwright visual, CI lint, code review) failed to catch. The defects are not edge cases. They are the elevator pitch of the project.

This retrospective treats that as the central lesson.

## 2. What Went Well

- **Three.js reverse-Z foundation (Story 1.5)** — `reversedDepthBuffer: true` plus the WorldGroup floating-origin pattern carried through every subsequent story without regression. Branded `WorldVec3`/`RenderVec3`/`MeshLocalVec3` types caught at least three implicit-conversion bugs at compile time.
- **VTRJ per-segment chunking (Story 1.4 pivot)** — the Option B choice to chunk per SPK segment rather than ship a single monolithic VTRJ paid off in Story 1.12. The 2.1M km SPK discontinuity that would have rendered as a visual seam was cleanly handled by separate chunk boundaries.
- **Lit 3.3 scaffold without decorators (Story 1.7)** — sidestepping the Vite/esbuild decorator gap by registering via `customElements.define()` saved an unknown number of hours of compatibility debugging.
- **Code review depth on a few specific stories** — Story 1.9 caught the simEt clamp-bypass on property write. Story 1.12 caught the trajectory-zeros-before-prefetch defect. Story 1.5 caught the precision-smoke lookAt bug. These were real saves.
- **Solar System Scope texture pivot (Story 1.13)** — the Björn Jónsson licensing block was caught and re-sourced without losing visual quality.
- **Manual verification, as a practice** — performing it once at end-of-epic exposed defects the entire test pyramid missed. The pyramid is not enough. This is the most important lesson of Epic 1.

## 3a. The Lesson Behind The Lesson — Brotli (discovered during retro execution, 2026-05-19)

While executing Story 1.15 during the retro itself, an attempt to manually verify the dev server in the user's local Chrome 148 (stable) hit `/unsupported.html?reason=brotli`. Diagnosis showed Chrome 148 genuinely lacks `DecompressionStream('br')` — and so does every other Chrome, because brotli was **never standardized into the Compression Streams API**. Only `gzip`, `deflate`, `deflate-raw` ship.

The Story 1.8 boot probe was designed against an assumption that brotli would land in browsers. It never did. The probe correctly identifies the missing feature and redirects; the architecture's assumption that VTRJ files would be client-side brotli-decompressed via the JS API is wrong.

**Voyager has never been reachable in any current production browser.** Every "manual verification" claim in Epic 1 was made through Chrome DevTools MCP with an `initScript` brotli shim. The shim was acknowledged in code-review comments and tracked in `deferred-work.md`, but its load-bearing role was not — it wasn't "an environmental quirk of MCP," it was "the only path to a running app, anywhere."

**This is the same root-cause class as the four HIGH defects** (the test pyramid passed everything; the deployed product was broken), except deeper: the entire MCP verification tier was also masking the brotli gap. The lesson the retro originally surfaced — "the test pyramid is necessary but not sufficient; manual smoke catches what tests miss" — needs an extension: **MCP-driven smoke is also a verification tier that can mask environmental assumptions.** A real-user-browser smoke at end-of-epic (or every story touching the boot path) was the missing layer.

**Resolution:** Story 1.16 was added to Epic 1 to switch from JS `DecompressionStream('br')` to HTTP-level `Content-Encoding: br` (Chrome's network stack handles this fine, just not the JS API). After 1.16 lands, Voyager works in real Chrome. Epic 1 does not close until 1.16 ships.

**Updated Decision C** (originally "Story 1.15 patches manual-verification defects within Epic 1"): now reads "Story 1.15 + Story 1.16. 1.15 patches the four HIGH defects; 1.16 fixes the brotli architectural assumption. Epic 1's 'done' state means 'works in a current production browser, end-to-end, without an MCP-only shim.'"

## 4. Other Surprises

- **Agent completion-message hygiene** — ~4–5 spawns out of 42 (≈10–12%) completed their work to disk but went idle without sending a structured `STATUS: completed` message. The lead had to issue ad-hoc status-check pings. Symptom of skill prompts not explicitly instructing the agent on the team-lead handoff protocol.
- **NFR-P6 (Story 1.10) is mathematically impossible as written** — "ClockManager mutations complete in ≤ 100µs P99" runs into the browser's wall-clock measurement floor (~1ms). QA worked around with a comment in `clock-manager.ts` and an entry in `deferred-work.md`.
- **NFR-M4 (Story 1.14) conflates "human-readable" with "lint-clean"** — cycle log TAB-separators are readable to humans but trip MD010. Worked around by entry in `deferred-work.md`.
- **Both NFR tripwires reveal a planning/implementation gap.** The PRD/architecture say one thing, the code (with comments) says another, and the deferred-work file is the only place the truth lives. New devs reading planning artifacts will not know which NFRs are "as-written" vs "as-implemented-with-comment."
- **toktx requires admin (Story 1.13)** — KTX2 texture pipeline blocked by Windows ACL on the toktx binary. Pivoted to PNG-2k for Epic 1; deferred KTX2 + 4k to Story 4.3.
- **Chrome DevTools MCP redirect to /unsupported.html?reason=brotli** — Chrome 148 in the MCP sandbox genuinely lacks `DecompressionStream('br')`. Bypassed for manual verification via `initScript` shim; this would have been a real problem if the MCP-driven Chrome were our verification tier.

## 5. Decisions

Locked in during this retro, ordered by leverage:

### Decision A — Add Chrome DevTools MCP regression as a test tier

Add a new validation tier driven by the `mcp__chrome-devtools-mcp__*` tools. Sub-agents (dev, QA, code-reviewer) must be configured to have these tools in their allowed tool set so they can navigate the dev server, interact with the deployed UI, and assert real DOM/rendering state. This tier catches all four manual-verification defects on the same PR they are introduced.

**Action items:**
- A1. Update `.claude/agents/` definitions for dev / QA / code-reviewer roles to include `mcp__chrome-devtools-mcp__*` in the allowed tools. Owner: Lead. Before Epic 2 starts.
- A2. Add a `/bmad-qa-generate-e2e-tests` skill output stage that runs a Chrome DevTools MCP smoke against the dev server: navigate → exercise the feature → assert observable state (DOM attributes, rendered transforms, console-clean). Owner: Lead. Story 2.0 work item.
- A3. Address the `DecompressionStream('br')` gap for the MCP-driven Chrome: either serve uncompressed bundles in MCP mode, or document the `initScript` shim as standard for any agent driving Chrome via MCP. Owner: Story 2.0.

### Decision B — Layered AC discipline (all three)

Apply all three structural fixes to story acceptance criteria:

1. **Mandatory "integration AC" in every service-introducing story.** Template: "Consumer `X` reads from this service and produces observable effect `Y`."
2. **Explicit `consumed-by` field linking stories.** A story introducing a service lists every later story that will consume it. Each consumer story inherits an AC to verify the wire-up.
3. **Manual smoke test as exit criterion.** Before any story moves to done, the lead spins up the dev server and exercises the feature in browser. Not optional.

**Action items:**
- B1. Amend `bmad-create-story` skill template to add `## Integration ACs` and `## Consumed-by` sections. Owner: Lead. Before Epic 2's first create-story.
- B2. Amend `bmad-code-review` skill to fail review when integration ACs are absent on service-introducing stories. Owner: Lead. Before Epic 2's first review.
- B3. Add "manual browser smoke" as final stage in `/epic-cycle` per-story flow, before commit. Owner: Lead. Before Epic 2's first story.

## 9. Closing Note

Epic 1 delivered the technical foundation — reverse-Z, floating-origin, VTRJ pipeline, Lit scaffold, ClockManager, HUD components, trajectory rendering, celestial bodies, CI/CD. The architecture is sound. The defects are integration defects, not foundation defects. Fixing them in Story 1.15 will not require revisiting any design decision; it will require wiring components that should have been wired the first time.

The most valuable artifact of Epic 1 may be this retrospective itself: it surfaced that the test pyramid is necessary but not sufficient, that AC discipline must address inter-story handoffs explicitly, and that one manual browser session at end-of-epic catches what no automated tier can. Epic 2 starts with those guardrails in place.
