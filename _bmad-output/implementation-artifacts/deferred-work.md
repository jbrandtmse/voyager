## Deferred from: code review of 1-4-bake-pipeline-scaffold-and-l1-python-validation-harness (2026-05-18)

- **[1.4 / LOW]** `bake/src/vtrj_writer.py` defines `_HEADER_STRUCT` twice (lines 51 and 57), with the first definition immediately overwritten by the second and a comment block explaining why. Same shape in `bake_trajectories.py` (no duplicate constant there but the imports/branching reads as half-finished refactor). **Why deferred:** Pure code hygiene; the runtime behavior is exactly correct (the second definition wins). **Suggested resolution:** drop the dead first assignment and the explanatory comment block; keep only `_HEADER_STRUCT = struct.Struct("<4sHiddId2s")`.

- **[1.4 / LOW]** `bake/src/vtrj_writer.py:read_vtrj` does not validate `body_id ∈ {-31, -32}` on read, even though `write_vtrj` validates it on write. The asymmetry means a corrupted-on-disk VTRJ with an out-of-set body_id would deserialize cleanly. **Why deferred:** Story 1.6 owns the runtime loader (TypeScript), which gets its own body-id guard; this Python `read_vtrj` is used today only by the validator, which already cross-checks `header["body_id"] == naif_id` against the manifest entry. **Suggested resolution:** add `if body_id not in ALLOWED_BODY_IDS: raise ValueError(...)` between the magic/version checks and the body parse.

- **[1.4 / LOW]** `bake/src/vtrj_writer.py:write_vtrj` computes the recorded SHA-256 on the in-memory `compressed` buffer (line 151), not by re-reading the just-written `.part` file. Story 1.3's `acquire_kernels.py` deliberately re-reads from disk before atomic-renaming so MITM bytes never appear at the canonical path. The bake's `write_vtrj` writes the bytes it generated locally (no network), so the threat model is different — but the consistency contract differs from Story 1.3. **Why deferred:** Storage-layer corruption between `write_bytes` and `replace` is rare; the line-147 size check is a partial guard. **Suggested resolution:** re-hash the `.part` file via `hashlib.sha256(part_path.read_bytes()).hexdigest()` and assert equality with the in-memory hash before `replace`.

- **[1.4 / LOW]** `bake/src/validate_l1.py:_furnish_kernels` re-furnishes the full kernel set including `ck` kernels (priority dict line 84 has `"ck": 5`), whereas `bake_trajectories.py:217` explicitly excludes `ck` via `kinds=("lsk", "pck", "fk", "sclk", "spk")`. The asymmetry is harmless (CKs aren't queried by `spkgeo`) but slows validator startup marginally and disagrees with the bake-side furnish set. **Why deferred:** CK kernels load fast (~tens of MB total); functionally inert. **Suggested resolution:** narrow the validator's furnish set to match the bake's: skip `ck` in `_furnish_kernels`.

- **[1.4 / LOW]** `bake/src/bake_trajectories.py:226` matches each spacecraft's SPK kernel by substring (`f"Voyager_{name[-1]}" in k.target_path`), which would match a hypothetical `Voyager_12.bsp` kernel. **Why deferred:** kernels-manifest.json is version-controlled; no such kernel exists. **Suggested resolution:** add an explicit `spk_match` field to the `BODIES` tuple or match on the kernel's `kind == "spk"` plus an exact-prefix check on `target_path`.

- **[1.4 / LOW]** `bake/src/manifest_writer.py:_git_head_sha` returns `"unknown"` on `subprocess.TimeoutExpired` (10s) silently, with no stderr breadcrumb. Anyone debugging "why does bakeCommit say 'unknown' on this CI run?" will not find a log line pointing at the timeout. **Why deferred:** Observability-only; doesn't affect correctness. **Suggested resolution:** emit a one-line `print(..., file=sys.stderr)` when falling back to `"unknown"` and include the failure mode (`timeout` / `not found` / `non-zero`).

- **[1.4 / LOW]** `bake/src/manifest_writer.py:emit_manifest` accepts an arbitrary `datetime` for `bake_timestamp` but `strftime("%Y-%m-%dT%H:%M:%SZ")` always appends `Z` regardless of the input's tzinfo. A caller passing a naive or non-UTC datetime would emit a false-UTC timestamp. **Why deferred:** the default code path uses `datetime.now(timezone.utc)` and the only test caller passes `tzinfo=timezone.utc`. **Suggested resolution:** if `ts.tzinfo is None`, raise; if `ts.tzinfo != timezone.utc`, convert via `ts.astimezone(timezone.utc)` first.

- **[1.4 / LOW]** `bake/src/bake_trajectories.py:183` `boundary_inset = min(0.01, span * 1e-9)`. For very short segments (sub-second), the inset falls below f64 ULP near ET values ~1e9 and is silently ignored. The Voyager merged SPKs have no such segments; this is purely a latent corner case. **Why deferred:** Not exercisable by the pinned kernels. **Suggested resolution:** when adding planet-trajectory baking in Story 1.13, add an assertion that `boundary_inset > eps(et_start)` to fail loud rather than silently inset to zero.

- **[1.4 / LOW]** Test `bake/tests/test_bake_defense.py:test_segments_do_not_overlap` sorts ranges before asserting non-overlap, which passes even if the manifest emits segments in a non-canonical order. The bake always emits in start-ET order, so this is belt-and-braces only. **Why deferred:** The bake's emission order is already deterministic. **Suggested resolution:** assert the manifest's per-body `files[]` is already sorted by `timeRangeEt[0]` *before* the overlap check.

- **[1.4 / LOW]** `justfile` recipes `ck-inventory` and `fk-inventory` (Story 1.3 helper wrappers) are not enumerated in `test_bake_defense.py:REQUIRED_RECIPES`. A future refactor could drop them without the test catching it. **Why deferred:** AC6's required-recipe list doesn't mention them; they're convenience wrappers. **Suggested resolution:** add them to `REQUIRED_RECIPES` when Story 1.13 wires up the planet kernels (where the FK inventory becomes load-bearing again).

## Deferred from: code review of 1-5-three-js-renderer-foundation-with-reverse-z-and-floating-origin (2026-05-18)

- **[1.5 / LOW]** `RenderEngine.setSize()` updates the renderer backing buffer and camera aspect, but does not re-call `setPixelRatio(window.devicePixelRatio)`. Dragging the window between monitors with different DPRs will leave the rendered pixel density stale until the page is reloaded. **Why deferred:** Single-monitor desktop is the v1 target tier; the rendering remains correct (just lower-resolution if dragged onto a HiDPI monitor). **Suggested resolution:** add `this.renderer.setPixelRatio(devicePixelRatio)` inside `setSize()`, gated on `typeof devicePixelRatio !== 'undefined'` to keep node-test parity.

- **[1.5 / LOW]** `RenderEngine.setCameraPosition(world)` stores the passed `WorldVec3` by reference (`this.cameraWorldPos = world`). A caller that re-uses a single `worldVec3` buffer and mutates its contents between frames would silently aliased-mutate the engine's recenter source. **Why deferred:** No current caller does this; the precision-smoke scene constructs a fresh `worldVec3` each frame. **Suggested resolution:** when Story 1.6's `EphemerisService` lands, decide whether to copy on entry (`this.cameraWorldPos = worldVec3(world[0], world[1], world[2])`) or document the borrow semantics. ADR-worthy if the per-frame allocation becomes a hot-path issue.

- **[1.5 / LOW]** `RenderEngine.tick()` allocates a fresh `Float32Array` via `floatingOriginOffset` every frame. At 60 FPS this is ~3.8 KB/min of garbage — trivial today but accumulates in long-running sessions and trips the GC. **Why deferred:** AC2 prescribes the per-frame recenter; the allocation is the price of the explicit boundary. Premature optimization. **Suggested resolution:** if Story 7.6's perf budget shows GC pressure, expose a pool / reuse buffer inside `floatingOriginOffset` (would require relaxing the "second allow-listed Float32 site" framing — fold into the same ADR as the borrow-vs-copy decision above).

- **[1.5 / LOW]** `GPUCapabilityProbe.run()` returns `supportsFloatDepth` based on `WEBGL_depth_texture` extension OR `DEPTH_COMPONENT32F` constant presence. The latter is a static numeric constant that exists on every WebGL2 context regardless of actual format support — so `supportsFloatDepth` is effectively `true` on every WebGL2 context. **Why deferred:** No downstream code consumes `supportsFloatDepth` yet; the field is informational until a renderer path conditionally uses 32-bit depth. **Suggested resolution:** when a consumer lands, switch to a renderable-format probe (e.g., create a depth renderbuffer with `DEPTH_COMPONENT32F` and check `gl.checkFramebufferStatus`).

- **[1.5 / LOW]** `precision-smoke.ts` constructs `BoxGeometry` and `MeshBasicMaterial` but does not dispose them on `engine.dispose()` (the `dispose` callback only calls `engine.dispose()`). The browser will GC them when the page unloads, but the Three.js best practice is explicit GPU resource disposal. **Why deferred:** Single-page dev-mode session; no leak in practice. **Suggested resolution:** add geometry/material disposal alongside the engine disposal when Story 7.x audits resource lifecycle.

- **[1.5 / LOW]** `web/index.html` retains the Vite default `<title>web</title>` (carried over from the Story 1.1 deferral). Story 1.5 lands the renderer but not the page chrome. **Why deferred:** UI shell is owned by Story 1.7 (Lit scaffold) per sprint plan. **Suggested resolution:** set the title to "Voyager" as part of Story 1.7's design-tokens + Lit scaffold work.

- **[1.5 / LOW]** `main.ts`'s `ensureCanvas` clears all children of the mount node, including any pre-existing styles or accessibility landmarks. The Vite scaffold puts the title and version banner in `#app`, so clearing is correct today — but a future Story might mount a landmark `<header>` before the canvas, and this code would silently delete it. **Why deferred:** No such landmarks exist yet. **Suggested resolution:** when Story 1.7 introduces the Lit shell, switch from "clear and append" to "find or create a dedicated canvas-host element" so other DOM children survive.

- **[1.5 / LOW]** `RenderEngine`'s internal wall-clock ET (`V2_LAUNCH_ET_SECONDS + (nowMs() - startTimeMs) / 1000`) advances continuously even while `stop()` has paused rendering. Resuming after a long pause yields a time jump. **Why deferred:** The placeholder ET is explicitly transient; Story 1.10's real `ClockManager` owns pause/resume semantics. **Suggested resolution:** remove this in Story 1.10 in favor of injecting the real `ClockManager`.

## Deferred from: code review of 1-10-v-play-button-simulation-clock-and-v-speed-multiplier (2026-05-18)

- **[1.10 / LOW]** `ClockManager.tick(...)` keeps running (and reclamping to `MISSION_END_ET`) after the mission end is reached; no internal stop.
