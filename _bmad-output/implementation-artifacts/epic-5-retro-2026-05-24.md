## What didn't go well

### 🚨 Top concern: Cross-review surfaced 4 defects the entire Story 5.X pipeline missed

After all 5 stories shipped clean (each with a passing lead-driven Chrome DevTools MCP smoke), a user-prompted cross-review surfaced:

1. **BUG-E5-007 (CRITICAL):** ALL production-built pages shipped with broken layout because the main CSS file was never linked from the HTML. Every HUD corner collapsed to overlapping rectangles at (0,0). The Story 5.0 BUG-006 regression spec PASSED against this state because it asserted on `<h2>` text not layout. The L4 Playwright suite (Story 4.9 + 5.4) PASSED because baselines were captured WITH the broken state — pixel-diff vs. self is clean.

2. **L4 stable-frame Condition 5:** The PBD substate baselines flaked at 2% pixel-diff because the canvas-fingerprint stability check only sampled the WebGL canvas, missing the Story 5.3 HTML composite layer. Pre-existing per Story 4.9 retro addendum item 3 ("L4 HUD region masking") but exacerbated by Story 5.3's new HTML overlay.

3. **BUG-E5-008 (MED):** HUD top-right icons (help, chapter-index) at `top: var(--v-edge-margin)` overlapped the HUD date / attitude / distance column at the same y. Invisible pre-BUG-007 (everything was collapsed to (0,0)).

4. **BUG-E5-009 + follow-up (MED):** Bottom-row scrubbers spanned full width, placing chapter-marker labels under the play button glyph and the speed-multiplier readout. Same dependency on BUG-E5-007.

**The root pattern:** none of the 5 stories' smoke gates exercised the production-build layout AT ALL. Every smoke either ran against dev mode (npm run dev + Chrome DevTools MCP) OR against the production build but asserted on TEXT not LAYOUT/POSITIONING. The L4 Playwright suite, which DOES run against the production build, locked the broken state in as baselines because all stories captured baselines while BUG-E5-007 was active.

**User-prompted second pass caught 2 more bugs.** The lead's initial cross-review caught BUG-E5-007 + the stable-frame issue. The user noticed text overlap in the upper-right and bottom areas; that prompted BUG-E5-008 + BUG-E5-009. Then the user noticed scrubber-vs-readout overlap on the right side; that prompted the BUG-E5-009 follow-up. So even the lead's cross-review pass was incomplete on the first try — three of the four bug fixes needed user attention to surface.

### Other concerns worth recording:

- **Story 5.0's AC1 BUG-006 regression spec didn't catch BUG-007 because it asserted on text.** The spec confirmed `<h2>` rendered "Pale Blue Dot" + the right slug — both passed because the title text rendered correctly even with the broken CSS (the H2 was just visually mispositioned). A LAYOUT-asserting test (e.g., assert the H2's `getBoundingClientRect().left >= someExpectedMinimum`) would have caught BUG-007 at Story 5.0's commit. Lesson for future Story-X.0 production-build regression specs: assert on positioning AND text.
- **Speedup-factor amendment surfaced the spec-vs-reality drift only at dev time.** Story 5.2's epic spec said "50× speedup" but the actual math (180s arc / 5 historical hours) is ~100×. The dev's Rule 5 amendment caught this — but the spec author should have computed it at planning time. Spec arithmetic that depends on a separate fact source (MISSION_FACTS.md timing) is a drift risk.
- **Story 5.1's `composite_active` substate semantic** was authored as "all six plates visible simultaneously" — directly contradicting the eventual Story 5.3 requirement ("at most ONE plate visible") AND the epic-spec 30-second-Earth-pause success criterion. Story 5.3 caught and amended it via Rule 5, but this is a forward-coherence gap: Story 5.1's substate definitions weren't validated against Story 5.3's eventual contract because Story 5.3's contract didn't exist yet. Forward coherence requires either (a) co-authoring closely-coupled substate definitions across Stories 5.X at planning time, or (b) treating Story 5.1's substate semantics as PROVISIONAL until consumed by 5.2/5.3.
- **Detail scrubber chapter-marker overlap is still unresolved.** 4 pairs of close chapters (V2L/V1L, V1J/V2J, V1S/V2S, V2N/PBD) render labels at overlapping x-coordinates on the multi-decade scrubber. BUG-E5-009 fixes the play-button-vs-scrubber overlap but doesn't address intra-scrubber clustering. Routed to Story 6.2 / Story 7.6 marker-clustering work.

### Epic 4 lessons that paid off in Epic 5

- **Rule 5 in-place amendments** — 3 inline amendments across Story 5.2 (mixed-coverage branch + speedup recomputation) and Story 5.3 (substates.ts composite_active repositioning). Same discipline as Epic 4's 6 amendments. Zero "code comment + deferred-work" workarounds.

- **Rule 11 build-pipeline E2E tests** — Story 5.3's `build_pbd_plates.test.ts` is a true E2E test against synthetic JPEG fixtures with idempotency + dominant-color-per-crop verification. The pattern Story 3.3 established carried forward cleanly.
- **Pattern reuse 4.5 → 4.6 → 4.7 → 5.1 foundation** — Story 5.1's Path A subscriber topology enabled Stories 5.2 + 5.3 to extend with zero new wiring topology. Same kind of "well-designed foundation → zero-cost extensions" payoff.

### New lessons from Epic 5

1. **The test pyramid passing + lead-driven Chrome DevTools MCP smoke passing IS NOT SUFFICIENT to detect broken production-build LAYOUT.** BUG-E5-007 shipped with 5/5 stories' smoke gates GREEN — every smoke either ran against dev mode (where CSS loads correctly via HMR) OR against production but asserted on text content (which renders correctly even with broken positions). The Lesson 8 closure ("test pyramid is necessary but not sufficient — per-story smoke is the bridge") needs a sub-lesson: **per-story smoke against PRODUCTION BUILDS must assert on LAYOUT (positions, dimensions, spatial relationships), not just on TEXT content.** This is action item #1.
2. **A baseline captured against a broken state passes its own regression check.** L4 Playwright tests for all 9 → 14 chapter baselines PASSED throughout Stories 4.9 + 5.4 because the baselines were captured WITH BUG-E5-007 active. Pixel-diff vs. self is clean when both sides are broken in the same way. Defense: at least one canonical reference (e.g., a small fixture HTML asserting "this is what the HUD top-right corner should look like at viewport 1280x720") that L4 can pixel-diff against.
