# Spacecraft models

3D models served by Vite at `/models/`. Loaded at boot via Three.js's
`GLTFLoader` with `MeshoptDecoder` (EXT_meshopt_compression, per ADR-0006)
and `KTX2Loader` (KHR_texture_basisu, per ADR-0006 Â§ Decision step 3)
registered. LFS-tracked via the `*.glb` pattern in the repository-root
`.gitattributes`.

## Voyager LOD chain (Story 3.3)

Story 3.3 replaced Story 1.12's single-LOD `voyager.glb` with a 4-level
`THREE.LOD` chain generated by `web/scripts/build_glb.ts`. The four LOD
GLBs (`voyager-lod0.<hash>.glb` through `voyager-lod3.<hash>.glb`) are
content-hashed, meshopt-compressed, and ship KTX2 textures via Basis
Universal supercompression. Each output GLB preserves the named
`BUS / SCAN_PLATFORM / HGA` hierarchy that downstream stories 3.4 / 3.5
/ 5.2 consume via `scene.getObjectByName(...)`.

### Asset pipeline (Story 3.3)

1. **Input** â€” `bake/inputs/models/voyager-raw.glb` (1.72 MB, Draco-compressed,
   3 WebP+PNG textures).
2. **Mesh mapping** â€” `bake/inputs/voyager-mesh-mapping.json` documents the
   bounding-box heuristic that re-parents the upstream flat mesh tree into
   the named `BUS / SCAN_PLATFORM / HGA` groups + the SCAN_PLATFORM hinge
   offset + the HGA orientation quaternion.
3. **Pipeline** â€” `web/scripts/build_glb.ts`:
   - Decode Draco (`KHRDracoMeshCompression`)
   - Restructure to named hierarchy per the mapping JSON
   - Strip `EXT_texture_webp` + `KHR_draco_mesh_compression`
   - **KTX2 transcode** via `toktx` (Khronos KTX-Software CLI):
     UASTC for baseColor textures, ETC1S for AO maps
   - `prune` â†’ `dedup` â†’ `weld` â†’ `meshopt` (EXT_meshopt_compression)
   - For each LOD level: clone + `simplify(ratio)` + write
4. **Output** â€” four LOD GLBs at `web/public/models/voyager-lod{N}.<hash>.glb`
   plus `bake/out/models-manifest-fragment.json` consumed by
   `bake/src/manifest_writer.py` to populate `manifest.models[]`.

### Prerequisites for the pipeline

The pipeline requires **`toktx`** (Khronos KTX-Software CLI) to be on PATH:

- **Install:** <https://github.com/KhronosGroup/KTX-Software/releases>
  (download a recent release matching your OS â€” pre-built binaries
  available for Linux, macOS, and Windows; or `brew install ktx` on macOS;
  or install via the apt-get path documented in `.github/workflows/ci.yml`
  for Ubuntu 22.04+).
- **Verify:** `toktx --version` should print a Khronos KTX-Software
  banner.

If `toktx` is not on PATH, `web/scripts/build_glb.ts` exits with a clear
error pointing at the install docs.

CI installs `toktx` automatically â€” see
`.github/workflows/ci.yml` Â§ "build-glb" job.

### Acquisition + regeneration steps

To rebuild the LOD chain after a fresh clone (or after updating the raw
input GLB):

```bash
# From the repository root:

# 1. Pull LFS-tracked binaries (raw GLB + future kernel data):
git lfs pull

# 2. Install toktx if you haven't already (see above).

# 3. Run the bake-glb pipeline:
just bake-glb

# This regenerates web/public/models/voyager-lod{0..3}.<hash>.glb +
# bake/out/models-manifest-fragment.json. After this, `just bake`
# (which chains bake-trajectories â†’ bake-attitude â†’ and reads the
# models fragment) produces a manifest.json with the populated
# `models[]` field.
```

If you wish to re-acquire the raw upstream NASA GLB (e.g., to check for
upstream updates):

```bash
curl -L -o bake/inputs/models/voyager-raw.glb \
  "https://raw.githubusercontent.com/nasa/NASA-3D-Resources/master/3D%20Models/Voyager%20Probe%20(B)/Voyager%20Probe%20(B).glb"

# Verify the hash matches THIRD_PARTY.md + bake/inputs/voyager-mesh-mapping.json:
sha256sum bake/inputs/models/voyager-raw.glb
# expected: bd86ded828dd3f459293aee4ffc3cd0998d8db67439317c8299650a1174c3289
```

If the SHA differs from THIRD_PARTY.md, update both files and
`bake/inputs/voyager-mesh-mapping.json` (`source_sha256` field) in the
same commit â€” the mapping is keyed off the input hash so a regenerated
input invalidates the mapping until the dev re-verifies it.

### Transition-window note

While the LOD chain is the production source of truth, `SpacecraftModels`
retains a legacy single-LOD fallback path (Story 1.12 pattern) so a fresh
clone where `just bake-glb` hasn't run yet doesn't crash at boot. The
fallback fetches `/models/voyager.glb` directly; in the Story 3.3 state
that file is removed (the LOD URLs replace it), so the fetch will 404
until `just bake-glb` runs and produces the LOD chain. The
`SpacecraftModels` loader logs `console.warn` once and continues â€” the
spacecraft simply doesn't render until the LODs are on disk and the
manifest is regenerated. See `web/src/render/spacecraft-models.ts` Â§
`resolveModelEntry` for the fallback logic.
